# Notes Summary (English, Detailed)

## 1. App Scope
- Pure HTML/CSS/JS version of the task list app.
- No backend. Data stored in the browser.
- Files live in: C:\academia\src\jp02_tasklist_html

## 2. Storage Decisions (Timeline)
- Started with localStorage for simplicity.
- Confirm page issue appeared when using file:// and module scripts.
- Switched to normal script tags to avoid file:// module restrictions.
- Migrated storage to IndexedDB for persistence and scale.

## 3. IndexedDB Persistence
- IndexedDB persists after closing and reopening the browser.
- Data is stored on disk (HDD/SSD) under the browser profile.

## 4. Why MySQL Needs Backend
- Browsers cannot connect directly to MySQL/Postgres.
- You need a backend/API or serverless function.
- Frontend-only options: localStorage, IndexedDB, File System Access API.

## 5. Chrome IndexedDB Location
Default profile path:
C:\Users\<your-username>\AppData\Local\Google\Chrome\User Data\Default\IndexedDB\

Folder naming rules:
- https sites: https_<host>_0.indexeddb.leveldb
- http sites: http_<host>_0.indexeddb.leveldb
- local files: file__0.indexeddb.leveldb (all file:// origins)

## 6. How To Identify The Right Folder
1. Open the app and add/edit a task.
2. Refresh the IndexedDB folder list in Explorer.
3. The folder with the latest timestamp is the active one.

## 7. DevTools Access (Chrome)
- F12 -> Application tab -> IndexedDB
- Database: tasklist_db
- Stores: todos, categories, state

## 8. File:// Script Note
- Some browsers restrict module scripts on file://
- App uses a normal script tag to avoid this issue.

## 9. Seed Data Behavior
- When IndexedDB is empty, it seeds default tasks/categories.
- If data exists, it reuses existing records.

## 11. Current Validation UX
- Required-field, percent-range, and date-order checks run on the Create/Edit form.
- Due Date must be on or after Start Date.
- Invalid controls receive a red highlight style.
- Focus moves to the first invalid field after submit.

## 12. Current Helper Guidance UX
- The Create/Edit form shows inline helper text for Start Date, Due Date, and % Complete.
- Task Name and Description show live character counts.
- Character hints update while the user types.

## 13. Current Confirm UX
- The Confirm screen groups review data into Basic Information, Schedule, and Progress.
- This grouping is intended to improve scanability before Save.

## 14. Current Empty / Success State UX
- The list view shows a guided empty state when filters return no rows.
- The empty state includes Reset Filters and Create Task actions.
- The Complete page shows a success banner above the saved record summary.

## 15. Current Table UX
- The list table uses a sticky header for easier scanning on long result sets.
- Sort indicators show active direction more clearly.
- Long task and assignee values are visually compacted to reduce layout breakage.
- Header labels are shortened on desktop to reduce width pressure.
- Delete is styled as a lighter destructive action while Edit remains primary.
- RAG is shown as a compact circular color marker without text.
- Pagination is fixed to 10 items per page with `前へ`, `次へ`, page links, and total/range summary.
- Due dates are highlighted red when overdue and yellow when due within 3 days.
- Bulk delete is supported with per-row checkboxes, select-all for visible rows, and confirmation.

## 16. Current Accessibility UX
- Keyboard focus is visually strengthened on major interactive controls.
- Form fields use linked helper and error descriptions.
- Validation errors remain announced through dedicated error containers.
